Executive Summary

The Speedtest-Validator codebase is a fully implemented, production-ready Python pipeline (~5,900 lines across 28 files) that automates T-Mobile DAS site RF validation. All 6 phases are wired end-to-end: Discovery → VLM Extraction → CIQ Correlation → Threshold Check → Knowledge Analysis → Excel Output. VRAM lifecycle management, Pydantic schema validation, retry loops, and sanitization are all in place. This is not scaffold — it is deployed code.

The evaluation uncovered 29 gaps across architecture, code quality, test coverage, and operational readiness. Of these, 5 are critical, 9 are high severity, and 15 are medium/low. A key architectural decision: the pipeline should support two output modes — --fast (Phases 1-4 only: VLM extraction + thresholds, no LLM analysis, Excel without Observations/Recommendations/KPI Impact columns) and --full (all 6 phases including LLM-powered analysis columns). Word document output (output_docx.py) has been deprecated. Beyond bug fixes, two strategic enhancements — a Causal DAG (JSON data already in place at code/knowledge/causal_dag.json) for root-cause reasoning and Qwen Agent for agentic VLM extraction — represent the intelligence ceiling upgrade.

C1: No Error Recovery on VLM Extraction Failure

CRITICALPIPELINE

The Gap

  • screenshot_parser.py:361-365 — When VLM extraction fails (ValueError), sm_data is set to None but processing continues. The result dict has service_mode=None, which propagates nulls through Phase 3-6.
  • main.py:128-151 — CIQ correlation blindly accesses sm.get("lte_params") on None service_mode, causing KeyError or producing all-zero threshold checks.
  • Impact: A single bad screenshot silently poisons an entire output row — the row appears in Output.xlsx with empty RF columns and a misleading "PASS" comment.

  • Retry with lower resolution: If first VLM call fails, retry with max_dimension=768 (lower quality but higher success rate with qwen3-vl).

S1: Causal DAG — Root-Cause Reasoning Engine

NEWSTRATEGICCAUSAL DAG

Current State: Flat, Isolated Lambda Triggers

  • kpi_mappings.py evaluates each condition independently: rsrp < -75 fires separately from sinr < 25, even though low RSRP causes low SINR which causes low throughput.
  • analysis_engine.py makes 3 independent LLM calls (observations, recommendations, KPI impact) with no shared reasoning chain. The model generates recommendations without knowing its own observations.
  • Result: If a cell has low RSRP AND low SINR AND low throughput, you get 3 separate KPI impact blocks that are really all describing the same root cause. No deduplication, no causal tracing, no root-cause isolation.
  • Physical Layer thresholds (C2) are loaded but never connected to RF parameters — VSWR→RSRP→SINR→throughput chain is invisible.

Target State: Directed Causal Graph

  • DAG nodes = measurable parameters (VSWR, RSRP, SINR, MCS, DL Throughput, KPI domains). Edges = causal mechanisms with explanatory text.
  • Backward traversal from symptom (DL fail) through causal chain to root (VSWR issue). Recommendations become specific and actionable.
  • Deduplication: Collapses correlated KPI impacts into one causal chain with one root cause and multiple downstream effects.
  • Predictive flagging: Identifies cells that are borderline on one parameter but already degraded on a downstream parameter — "one event away from failure."

Example Causal Chain: Physical Layer → KPI Failure

VSWR > 2.1 → Return Loss ↑ → RSRP Degraded → SINR Degraded → MCS Reduced → DL Throughput ↓ → CAP KPI FAIL
PCI Collision → SINR Degraded → BLER ↑ → RET KPI FAIL

S2: Deterministic Extraction Hardening + Optional --investigate Mode

REVISEDSTRATEGICEXTRACTION

Why NOT Agentic Tool-Calling for Core Extraction

  • Strategic contradiction: The Causal DAG moves intelligence from inference-time to build-time (precomputed reasoning paths). Agentic extraction does the opposite — gives the model a toolbox and says "figure it out step by step." These philosophies conflict.
  • Local model reliability: Tool-calling on qwen2.5vl:7b is materially weaker than frontier models. Every agent "step" is a full inference pass adding 30-60s of latency, and tool-call JSON formatting adds another failure mode on top of the existing ~75% first-pass rate.
  • Latency explosion: 48 screenshots × 2-4 agent iterations × 30s each = 1.5-4 hours vs ~25 min for single-shot. On 16GB VRAM with sequential model unloading, this is prohibitive.
  • Every proposed tool has a deterministic alternative that's faster, cheaper, and 100% reliable. See decision matrix below.

Target State: Deterministic Pre/Post-Processing + Optional Agent

  • Image preprocessing (replaces crop_and_zoom): Split Samsung screenshots into LTE/NR panels before VLM call using known layout coordinates. 20-line Pillow function, zero inference passes.
  • Post-extraction CIQ validation (replaces cross_reference_ciq): After VLM extraction, compare PCI/EARFCN/band against CIQ in Python. Flag mismatches. Already partially in Phase 3.
  • Bounds-based confidence scoring (replaces flag_ambiguous): If extracted value outside physical bounds (RSRP > 0, SINR > 50, negative bandwidth), flag automatically in Pydantic validator. No LLM needed to "decide" something is ambiguous.
  • Optional --investigate mode: A separate CLI mode where field engineers ask open-ended diagnostic questions ("why does this n25 cell have high RTWP when VSWR is clean?") using the DAG + LLM. This is where agentic reasoning genuinely adds value — as an interactive diagnostic tool, not the core pipeline.
