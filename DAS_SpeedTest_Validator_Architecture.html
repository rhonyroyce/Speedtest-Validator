DAS Speed Test Validator

Automated RF Validation Pipeline Using Local Vision-Language Models

Executive Summary

Manual DAS validation requires 2-3 hours per site: extracting RF parameters from screenshots, correlating with CIQ, synthesizing observations into Excel and Word. Errors are high, quality inconsistent. The DAS Speed Test Validator automates this using qwen3-vl:8b (local Ollama) for RF extraction, a deterministic knowledge engine (Python dicts, no vector DB), and automated deliverable generation. Runtime: ~15 minutes per site. Fully local, zero hallucination.

This report is the architecture blueprint. Includes 5 technical sections, 3 cross-sections, and 10 sequential Claude Code prompts that scaffold the entire system. Execute prompts in order to build the complete validator.

Architecture Overview

8 core modules plus orchestrator perform end-to-end RF validation automation: screenshot extraction (VLM), CIQ correlation, knowledge injection, analysis, and deliverable generation.

Core Modules

  • screenshot_parser.py: Discovers screenshots, calls qwen3-vl:8b, validates JSON schema, retries on failure
  • ciq_reader.py: Parses CIQ Excel (eUtran Parameters, gUtranCell sheets), extracts BW/PCI/EARFCN/MIMO
  • threshold_engine.py: Multi-dimensional MOP pass/fail: Service Mode (RSRP/SINR/RSRQ/TX) + Speed Test lookup from DAS_Validation_Thresholds.xlsx keyed by (SISO/MIMO × BW combo × Connection Mode: LTE/NR/EN-DC/NR-DC). Loads thresholds from Excel at runtime — never hardcoded.
  • knowledge_engine.py: Loads 4 Python dict modules (rf_parameters, throughput_benchmarks, kpi_mappings, mop_thresholds). No vector DB.
  • analysis_engine.py: Correlates RF with CIQ, calls gpt-oss:20b to generate Observations/Recommendations/KPI Impact. Anchored to KB.
  • ollama_client.py: Unified Ollama client. Supports qwen3-vl:8b (vision, ~5GB) and gpt-oss:20b (analysis, ~12GB). CRITICAL: 16GB VRAM — must unload one model before loading the other. Pipeline runs in 2 phases: (1) load VLM, extract ALL screenshots, unload; (2) load analysis LLM, generate ALL narratives, unload. Uses POST /api/generate with keep_alive:0 to unload.
  • output_xlsx.py: Generates Output.xlsx: Tab 1 = 16-column validation results (adds Connection Mode column), Tab 2 = threshold reference tables copied from DAS_Validation_Thresholds.xlsx
  • output_docx.py: Generates RF_Throughput_Analysis.docx (80+ paragraphs, multi-section, styled)

Vision Pipeline: qwen3-vl:8b

Service Mode Extraction: RSRP (-140 to 0 dBm), RSRQ (-20 to 0 dB), SINR (-20 to 30 dB), UE TX Power, measurement duration, cell ID (PCI/EARFCN/band).

Speed Test Extraction: DL Throughput (Mbps), UL Throughput (Mbps), latency (ms), test duration (sec), technology (LTE/NR/mixed).

JSON Schema Enforcement: Strict Pydantic schema. On validation failure: retry with corrected prompt (max 2 retries). Confidence: Pass 1 = 0.95, Pass 2 = 0.80, failure = 0.5 (flag for manual).

Anti-Hallucination: "If value not visible, return null. Do not invent." "RSRP must be -140 to 0. If outside, re-inspect." "Output ONLY JSON."

Knowledge Engine: Direct Extraction, Not RAG

~200 RF parameter-to-action mappings extracted from MOP PDF, RF analysis docx, KPI Excel. Stored as Python dicts in knowledge/ directory. Injected into prompts at analysis time. Zero vector DB complexity.

VERDICT: Direct Structured Extraction wins decisively. ~200 finite, well-structured mappings are better served as Python dicts than vector embeddings. RAG adds latency (260ms per query vs 1ms dict lookup), complexity (chromadb, embeddings, retrieval), and retrieval risk without compensating benefit.

4 Knowledge Modules:

  • rf_parameters.py: RSRP/SINR/RSRQ ranges → quality labels (excellent/good/fair/poor). Observations by (rsrp, sinr, rsrq) tuple.
  • throughput_benchmarks.py: (tech, bw, mimo, ca_layers) → theoretical DL/UL Mbps (3GPP + measurement data).
  • kpi_mappings.py: RF issue → {kpis: [list], severity, description}. RF degradation → service metric impact.
  • mop_thresholds.py: MOP pass/fail: RSRP (-75 to -50), SINR (>25), RSRQ (-12 to -3), DL/UL per tech/bw/mimo.

Output Generation

Output.xlsx (Tab 1 — 16 columns): BTS, Tech/Sector, Connection Mode (LTE Only/NR SA/EN-DC/NR-DC — auto-detected), Bandwidth, PCI, RSRP, RSRQ, SINR, UE TX Power, SM-ST Duration, DL Throughput, UL Throughput, Comment (PASS/FAIL), Observations, Recommendations, Impact on KPIs. Row count is DYNAMIC — equals number of cells tested (sectors × technologies). Tab 2: Threshold reference tables (SISO + MIMO + Service Mode) appended from DAS_Validation_Thresholds.xlsx for auditability.

RF_Throughput_Analysis.docx: Executive Summary (3 paras), Site Overview KPI table, Per-Cell Analysis (Heading 3 per BTS, 80+ paras total), Summary & Actions, Appendix (full 15-column table). Styling: H1/H2/H3 hierarchy, justified text, alternating table rows, footer ("Confidential | T-Mobile DAS | Date").

6-Layer Quality System

  • L1 Knowledge Injection: Thresholds from MOP/RF analysis embedded as Python dicts. All decisions trace to source docs. No hallucination about arbitrary thresholds.
  • L2 Multi-Pass Analysis: On Observation/Recommendation generation failure, retry with corrected prompt. Reduces incomplete narratives.
  • L3 JSON Schema Enforcement: All VLM outputs match strict Pydantic schema. Invalid or out-of-range values trigger retry. 100% structured, zero malformed data downstream.
  • L4 Quality Gates: Confidence thresholds. If confidence < 0.7, flag for manual review with high priority. Transparent tracking.
  • L5 Prompt Optimization: Prompts include few-shot examples, anti-hallucination guardrails, domain-specific instructions. Regular refinement in Claude Code.
  • L6 Sequential Model Loading (16GB VRAM): Vision tasks use qwen3-vl:8b (~5GB); analysis tasks use gpt-oss:20b (~12GB). Cannot run simultaneously — 16GB total VRAM. Pipeline phases: Phase 1 loads VLM, processes all screenshots, unloads via keep_alive:0. Phase 2 loads gpt-oss:20b, generates all narratives, unloads. GET /api/ps verifies unload before next load.

Quality Metrics: Extraction accuracy >0.9 (95%), schema compliance >98%, manual review <5%, narrative coherence >90%, deliverable integrity 100%.

Cross-Sections & Implementation

Cross-Section 1: RAG vs Direct — Why Direct Wins

DimensionVector DB RAGDirect Extraction
SetupHigh: chromadb, embeddings, retrieval pipeline, tune model, manage DBLow: Python dicts, no external deps
LatencyMedium: embed + retrieve + generate ≈ 260ms/queryLow: dict lookup ≈ 1ms
AccuracyRisk: wrong chunk retrieval possibleDeterministic: exact match on RF state
MaintainMedium: re-embed on doc changes, pipeline disruptionLow: edit Python dict, done
QA/AuditHard: difficult to inspect retrieved chunksFull: exact knowledge visible in code, trace all decisions
DepsMany: chromadb, sentence-transformers, numpy, scipy (~200MB)None: pure Python
Best ForLarge unstructured knowledge (wiki, news, support docs)Focused structured domain knowledge (~200 mappings)
VERDICT: For DAS validator, Direct Structured Extraction wins decisively. The knowledge base is finite (~200 RF parameter-to-action mappings extracted from MOP, RF analysis, KPI Excel). These are perfectly suited to Python dicts. RAG adds latency, complexity, and retrieval risk without compensating benefit when domain knowledge is finite and well-structured.
